## Finding 1: I1 atomicity invariant — VERIFIED SAFE

- **Severity:** CRITICAL (as scoped) — **Status: no vulnerability found**
- **Confidence:** 10/10 (read every call site)
- **File:** `bifrostd/src/orders/engine.ts:338-358` (`settleIncoming`), `:230-257` (`OUTGOING_SETTLED` handler)

`grep -n "settleHoldInvoice\|\.hold\.settle"` across `bifrostd/src/` shows
exactly **one** call site: `engine.ts:350`, inside `settleIncoming()`. Before
that line, two independent gates must both pass:

1. `order.state !== "OUTGOING_SETTLED"` → refuse (state-machine gate, `engine.ts:341-344`)
2. `!preimage || !hexPreimageMatches(preimage, order.paymentHash)` → refuse (cryptographic gate, `engine.ts:346-348`), where `hexPreimageMatches` (`engine.ts:51-53`) computes `sha256(preimage)` and compares to the stored hash.

The one place that sets `order.outgoing.preimage` (`engine.ts:249-255`,
inside the `OUTGOING_SETTLED` case) already re-validates
`hexPreimageMatches(event.preimage, order.paymentHash)` at `engine.ts:245`
before persisting it — so by the time `settleIncoming` reads
`order.outgoing.preimage`, it has already passed the hash check once, and
is checked again before use. No path settles the incoming leg without a
hash-verified preimage. **No fix needed.**

## Finding 2: I2 expiry invariant — VERIFIED SAFE

- **Severity:** CRITICAL (as scoped) — **Status: no vulnerability found**
- **Confidence:** 10/10
- **File:** `bifrostd/src/orders/engine.ts:94-117` (creation-time check), `:303-316` (`dispatchOutgoing` re-check), `sdk/src/expiry.ts:44-53` (`expiryInvariantHolds`)

**Correction to the audit scope:** `bifrostd/src/guard/` does not exist as
a directory — the invariant is enforced inline in `orders/engine.ts`, using
`expiryInvariantHolds` imported from `bifrost-sdk` (`sdk/src/expiry.ts`),
not a separate guard module. Enforced at three points:

1. **Order creation** (`engine.ts:103-117`) — throws `EXPIRY_INVARIANT_VIOLATION` before any state is created if the invariant fails.
2. **Immediately before outgoing dispatch** (`engine.ts:309-316`, PROTOCOL §6's required re-check) — refunds if the gap has closed since creation.
3. **Periodic sweep** (`engine.ts:192-199`) — catches orders that go stale waiting in `INCOMING_HELD`.

`expiryInvariantHolds` (`sdk/src/expiry.ts:44-53`) converts Lightning CLTV
correctly asymmetric per PROTOCOL §6: outgoing side uses
`OUTGOING_MS_PER_BLOCK = 600_000` (slow-block pessimism, overestimates how
long the outgoing leg can hang), incoming side uses
`INCOMING_MS_PER_BLOCK = 300_000` (fast-block pessimism, underestimates the
incoming hold window) — both directions shrink the apparent safety gap,
never widen it. **No fix needed.**

## Finding 3: Demo endpoints fail-open — ALREADY RESOLVED

- **Severity:** HIGH (as scoped) — **Status: fixed in this branch prior to this audit** (`/review`, same session)
- **Confidence:** 9/10
- **File:** `bifrostd/src/api/server.ts:53`, `:223`, `:262`

Originally gated on `NODE_ENV === "production"` (fail-*open*: reachable by
default, closed only if an operator remembered to set `NODE_ENV`). Fixed to
`DEMO_ENDPOINTS_ENABLED = process.env["DEMO_ENDPOINTS_ENABLED"] === "true"`
— **fail-closed by default**, both `handleDemoInvoice` and `handleDemoPay`
check `if (!DEMO_ENDPOINTS_ENABLED) return 404`. `handleDemoPay` was the
sharper edge here: it directs the client-side FNN node to pay any
caller-supplied invoice with no check that it belongs to a tracked order —
a fund-drain primitive if left open. `deploy/docker-compose.dev.yml`
explicitly opts in for the local dev/demo stack only. Verified: 404 by
default, 200 only with the explicit flag set.

## Finding 4: No authentication on any bifrostd endpoint — auth removed, HTLC crypto is the boundary

- **Severity:** HIGH (original finding) — **superseded 2026-07-17**
- **Confidence:** 9/10 (original finding stands as a historical record below)
- **Status:** Bearer-token auth was added, then deliberately removed.

**History, for the record:** this was originally found exploitable — a
live external `bifrostd` instance, reached via a public ngrok tunnel with
zero credentials, answered `GET /v1/health` and would have answered
`GET /v1/orders` (lists all orders: invoice strings, amounts, payment
hashes) and `GET /v1/inventory` (live liquidity) the same way. A full
`Authorization: Bearer <key>` gate was added on every route (HTTP + WS) to
close that.

**Auth removed 2026-07-17.** Decision: the HTLC cryptography is the real
security boundary here, not a bearer token in front of it. A quote expires
in 30 seconds. Creating an order requires actually **paying** the Fiber
hold invoice — no payment, no swap, the hub loses nothing. Bearer auth
added friction without meaningfully protecting the money path for a demo
hub, so it's gone: `bifrostd/src/api/server.ts`'s auth check and WS
`verifyClient` gate, `index.ts`'s `apiKey()`, and every `API_KEY` reference
in `deploy/docker-compose.dev.yml` and `demo/src/config.js`/`api.js` were
removed. Every route is reachable with no credential again.

**Residual gap, called out honestly rather than silently dropped:** the
original exploit scenario — anyone who can reach the port can run
`GET /v1/orders` to enumerate every order this hub has processed (invoice
strings, amounts, payment hashes, timestamps) and `GET /v1/inventory` to
read live liquidity — is back. Neither of those is guarded by the "no
payment, no swap" argument, since they're pure information reads, not
order-creation or payment paths. Accepted for a demo hub with no real
funds; matches `docs/STATUS.md`'s original "single trusted operator,
localhost-bound" framing (which itself has previously proven not to hold
in practice, once, when tunneled — see history above). If this hub is
tunneled publicly again, that read-only exposure is real, not
theoretical.

Verified: `curl http://localhost:8391/v1/health` → 200 with no
`Authorization` header. `POST /v1/quotes` → signed quote back, no header.

## Finding 5: Quote signature verification — VERIFIED SAFE

- **Severity:** MEDIUM (as scoped) — **Status: no vulnerability found**
- **Confidence:** 9/10
- **File:** `sdk/src/client.ts:71-84` (`Bifrost.getQuote`), `demo/src/pages/Checkout.jsx:56-61`

`Bifrost.getQuote()` (the SDK's own fetch-a-quote method) calls
`verifyQuote(quote, request, ...)` unconditionally at `client.ts:82`,
before returning — there is no way to obtain a `Quote` object through the
SDK without it passing the full PROTOCOL §9 checklist first.
`payAnyInvoice()` (`client.ts:123-159`) only ever gets its quote via this
same `getQuote()`, so it inherits the same guarantee.

The demo frontend (`demo/src/api.js`) talks to bifrostd with raw `fetch`
rather than the `Bifrost` class, so it does NOT get this for free — but
`Checkout.jsx:59` calls `verifyQuote(q, request)` manually immediately
after `getQuote()` and before advancing to the `"quoted"` UI phase; the
`Confirm & Pay` button that triggers `createOrder` is only reachable from
that phase. No unverified quote reaches `createOrder` in either path.
**No fix needed.**

## Finding 6: CORS policy — permissive, no production distinction

- **Severity:** MEDIUM
- **Confidence:** 8/10
- **Status:** VERIFIED
- **File:** `bifrostd/src/api/server.ts:78` (`json()` helper), `:299` (OPTIONS preflight handler)

`Access-Control-Allow-Origin: *` is hardcoded on every response, including
the preflight `OPTIONS` handler — no environment-based restriction exists.
Combined with Finding 4 (no auth), this means any website a victim's
browser visits can script requests to the hub on the victim's behalf (the
requests carry no session/cookie the hub would trust anyway, since there's
no auth — so the practical incremental risk beyond Finding 4 is small
today, but it removes a defense-in-depth layer once auth is added, since
`Access-Control-Allow-Origin: *` is incompatible with credentialed
requests).

**Recommendation:** once Finding 4's API key exists, CORS should stop
being `*` for the credentialed/key-bearing routes. For the current
no-credential model this is lower urgency; left as MEDIUM, not
auto-applied — folds into the same decision as Finding 4 (an allowlist
needs a real value: the demo's deployed origin, which doesn't exist yet
since `demo/` isn't deployed).

**Update — Finding 4 is now fixed with a bearer token, not cookies.**
Left `Access-Control-Allow-Origin: *` unchanged: a bearer token sent via
an explicit `Authorization` header (not `credentials: 'include'`/cookies)
isn't exposed by permissive CORS the way cookie-based auth would be — a
malicious page's script still can't produce a valid `Authorization` header
without already knowing the key. Still MEDIUM and still worth tightening
once `demo/` has a real deployed origin to allowlist; not urgent given
Finding 4's fix.

## Finding 7: No rate limiting — known, accepted gap

- **Severity:** LOW
- **Confidence:** 8/10
- **Status:** VERIFIED, pre-existing documented gap
- **File:** `bifrostd/src/api/server.ts:104` — `RATE_LIMITED` is a defined HTTP-status mapping for an error code that is never actually thrown anywhere in the codebase (`grep -rn "RATE_LIMITED" bifrostd/src` shows only the dead mapping and the protocol type definition).

**Exploit scenario:** an attacker can call `POST /v1/quotes` in an
unbounded loop. Each call is priced against live liquidity
(`fiberLiquidity`/`lightningLiquidity`) but doesn't reserve it — quotes are
non-binding until `POST /v1/orders` redeems one — so this is a request-volume
nuisance against the hub's own nodes' RPC, not a liquidity-drain vector by
itself (matches the `/cso` skill's own DoS/resource-exhaustion hard
exclusion — not reported as a standalone finding under normal scoring,
kept here only because it was explicitly in scope).

**Recommendation:** per-IP or per-key token bucket on `POST /v1/quotes`
before Milestone 2 goes anywhere near a non-demo deployment. Already
tracked informally via `docs/STATUS.md`'s "no per-key rate limits" note —
no new action needed beyond what's already accepted.
